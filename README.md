# AI全栈开发工作流

## 📖 简介

**本项目属于一个AI技能，用于快速创建一个AI全栈开发项目。**
AI全栈开发工作流是一套基于多 AI Agent 协作的**契约驱动全栈开发框架**。它通过定义需求分析师、架构师、前端开发者、后端开发者等 AI Agent 角色，结合标准化的 Rules、Skills 和模板，形成一套从需求到交付的结构化开发方法论。

**核心理念**：需求分析 → 架构设计（契约优先）→ 任务分配 → 前后端并行开发

## ⭐ 核心特性

- **多 Agent 协作** — 需求分析师、架构师、前端开发者、后端开发者各司其职，形成完整的 AI 开发团队
- **契约驱动开发** — 先定义 API 接口契约和数据库契约，前后端基于契约并行开发，杜绝耦合
- **结构化任务管理** — 任务看板 + 状态追踪（`todo` → `in-progress` → `wait-test` → `done`，异常路径 `blocked` / `failed`）
- **模块化需求管理** — 需求按模块拆分，变更日志完整记录，可追溯可回溯
- **架构师思维规范** — 贯彻单一职责、契约优先、数据独立性、可演进性四大原则

## ⚠️ 缺陷与限制

- **Agent 不会自动链式触发** —— 需求分析师 → 架构师 → 前端/后端开发者各阶段之间需要用户手动调用，工作流不会自动衔接。这保证了每个环节可审查、可回退，但也意味着开发节奏取决于用户的调度，无法做到"一次输入、自动产出"。
- **关键步骤需要人工介入** —— 数据库的创建与 schema 变更、契约文件的确认、跨空间路径配置等动作均需用户手动完成。Agent 仅生成中立 schema 描述与变更日志，DDL 执行、迁移脚本落地、环境切换由开发者自行处理。
- **契约前置依赖强** —— 前后端并行开发的前提是 API 与数据库契约已由架构师产出。若需求尚未稳定即进入开发，契约反复变更会引发前后端同步成本陡增，反而抵消并行优势。
- **三空间通过配置文件松耦合** —— dev-manager 与前端、后端空间之间仅靠 `setting.local.json` 中的路径互相引用。路径配置错误、目录迁移或多人协作未同步配置时，Agent 会找不到契约或任务文件，且缺少自动校验。
- **任务粒度按模块变更切分** —— 架构师按"一次模块变更 = 一个前端任务 + 一个后端任务"拆分，粒度偏粗。对于涉及多接口/多页面的大变更，开发者需在 Agent 内部自行进一步拆解，工作流本身不强制细粒度任务。
- **缺少自动化质量闭环** —— 工作流产出 `test.md` 测试用例文档，但不绑定 CI/CD、不自动运行测试、不自动校验契约与实现的一致性。测试通过与否、契约偏差检测仍依赖人工或外部工具。
- **变更可追溯但不可自动回退** —— 模块与数据库变更日志完整记录历史，便于追溯，但工作流不提供"按 change-log 回退到某版本"的能力，回退操作需人工对照日志逐步执行。

## 🤖 AI Agent 角色

| Agent | 职责 | 核心能力 |
| ----- | ---- | -------- |
| 🟥 **需求分析师** | 需求分析与模块拆分 | 将产品需求细化为结构化模块需求文档，定义验收标准 |
| 🟩 **架构师** | 架构设计与契约定义 | 程序功能设计、API 接口契约、数据库建模、开发任务分配 |
| 🟦 **前端开发者** | 前端功能实现 | 任务驱动的页面/组件/状态管理开发，严格遵循 API 契约 |
| 🟦 **后端开发者** | 后端功能实现 | 任务驱动的 API/业务逻辑/数据访问开发，数据库结构对比 |

## 📁 项目目录结构

下面是一个使用该技能生成的标准的项目目录结构：

```text
/
├── README.md                    # 项目简介，包含当前版本、项目背景等信息
├── setting.local.json           # 个人设置，每个开发者单独进行配置
├── requirements/                # 需求管理目录
│   ├── docs/                    # 通用需求文档目录，存放PRD、原型图、质量管理等文档
│   ├── module/                  # 模块级需求目录
│   │   ├── catalog.md           # 需求模块目录
│   │   ├── <模块名称>/           # 单个模块需求文件夹
│   │   │   ├── README.md        # 需求模块文档
│   │   │   └── change-log/      # 变更日志
│   │   │       └── YYYYMMDD-N.md
│   │   └── ...
│   └── ...
├── architecture/                # 架构设计目录
│   ├── README.md                # 架构设计说明
│   ├── module/                  # 程序功能模块
│   │   ├── catalog.md           # 程序功能模块目录
│   │   ├── <模块名称>/           # 单个程序模块
│   │   │   ├── frontend.md      # 前端功能文档
│   │   │   ├── backend.md       # 后端功能文档
│   │   │   ├── test.md          # 功能测试用例
│   │   │   └── change-log/      # 变更日志
│   │   └── ...
│   └── contracts/               # 架构契约
│       ├── apis/                # API接口契约目录
│       │   ├── catalog.md       # API接口目录索引
│       │   ├── interfaces/      # 接口定义目录
│       │   └── params/          # 出入参定义
│       └── database/            # 数据库设计
│           ├── catalog.md       # 数据库设计目录索引
│           ├── *.json           # 表/集合 schema 中立描述（JSON，按目标数据库映射为 DDL 或 schema 验证）
│           └── change-log/      # 数据库变更日志（按表名分文件，倒序记录该表所有 schema 变更）
│               └── <表名>.md    # 单表变更历史，与模块 change-log 通过 dev-task ID 双向引用
├── dev-tasks/                   # 开发任务拆分目录
│   ├── frontend.md              # 前端开发任务清单
│   └── backend.md               # 后端开发任务清单
└── templates/                   # 模板目录（模板、参考资料）
    ├── api-contract-template/
    ├── database-contract-template/
    └── 开发任务看板模板.md
```

## 🔄 开发工作流

```text
用户需求
  │
  ▼
需求分析师 ──► 需求分析、模块拆分，输出结构化需求文档
  │
  ▼
架构师 ──────► 程序功能设计、API 契约定义、数据库建模
  │              │
  │              ▼
  │           任务分配（前端任务 + 后端任务）
  │              │
  ▼              ▼
前端开发者 ◄───契约───► 后端开发者
  │              │
  ▼              ▼
  并行开发，基于契约各自实现
```

### 📋 各阶段说明

| 阶段 | 执行者 | 输入 | 输出 |
| ---- | ------ | ---- | ---- |
| 需求分析 | 需求分析师 | 用户需求描述 | [`requirements/module/`](requirements/module/) 下模块需求文档 |
| 架构设计 | 架构师 | 需求文档 | [`architecture/`](architecture/) 下程序功能设计、API/数据库契约 |
| 任务分配 | 架构师 | 架构设计结果 | [`dev-tasks/`](dev-tasks/) 下前后端任务清单 |
| 前端开发 | 前端开发者 | 前端任务 + 契约 | 前端代码实现 |
| 后端开发 | 后端开发者 | 后端任务 + 契约 | 后端代码实现 |

## 🚀 快速开始

### ✅ 前置条件

- 已安装 Claude Code（或兼容 Claude Code Skill 机制的 AI 编码助手）
- 已将本仓库作为 Skill 安装到助手可用技能列表中（克隆到 `~/.claude/skills/ai-fullstack-dev/`，或通过插件机制安装）
- 已创建一个空目录作为 AI 全栈开发项目的根目录（即 dev-manager 空间）
- 前端项目代码目录与后端项目代码目录已就绪（可空仓库，但路径需确定）

### 🛠️ 初始化项目

1. 在项目管理目录下调用 `ai-fullstack-dev` Skill 启动初始化
2. 按提示确认当前目录作为项目根目录，并输入前端项目路径 `$frontendProjPath` 和后端项目路径 `$backendProjPath`
3. 系统将自动创建目录结构、AI Agent 配置、Rules 和 Skills，并在前后端空间间复制共享技能 `task-driven-dev`
4. 检查项目根目录与前后端空间下生成的 `setting.local.json` 是否正确（结构参见 `templates/开发管理设置模板.md`）—— 三空间之间的路径互引全靠此文件，配置错误会导致 Agent 找不到契约或任务

### 💻 开始开发

1. 调用 **需求分析师** Agent，输入产品需求，生成结构化需求文档。例如：

``` prompt
读取 @requirements/docs/README.md 并输出模块需求文档
```

   产出位置：`requirements/module/<模块名称>/README.md`

2. 调用 **架构师** Agent，基于需求进行架构设计和任务分配。例如：

``` prompt
读取所有模块需求文档，并进行架构设计
```

或者只针对单个模块：

``` prompt
对 登录鉴权 需求模块进行架构设计
```

   产出位置：`architecture/module/<模块名称>/`（功能文档 + 契约 + 测试用例）与 `dev-tasks/`（任务清单）

3. 分别调用 **前端开发者** 和 **后端开发者** Agent，按任务清单并行开发。例如：

``` prompt
开始 登录鉴权-20260708-0 任务的开发
```

   产出位置：前端任务落到 `$frontendProjPath` 对应代码，后端任务落到 `$backendProjPath` 对应代码；任务状态同步写入 `dev-tasks/frontend.md` 或 `dev-tasks/backend.md`

4. 开发过程中通过 `dev-tasks/frontend.md` 与 `dev-tasks/backend.md` 任务看板追踪进度（状态流转：`todo` → `in-progress` → `wait-test` → `done`）
