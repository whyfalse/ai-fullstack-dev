name: 开发任务分配
description: 架构师完成程序功能设计后，可以调用此技能为前/后端开发者分配任务。
---

## 标识符双层概念

- **变更ID**（变更级）：`<任务简介>-<YYYYMMDD>-<序号>`，例如 `新增登录功能-20260520-0`。
  - 同一次模块变更的前后端任务共享同一个变更ID。
  - `<YYYYMMDD>-<序号>` 段与 `architecture/module/<模块>/change-log/<YYYYMMDD>-<序号>.md` 文件名完全一致，保证变更可追溯到设计变更。
  - 用于 change-log 之间、DB change-log、废弃横幅等**变更级**交叉引用（不带 fe/be 后缀）。
- **任务编号**（任务级）：`<变更ID>-<fe|be>`，例如 `新增登录功能-20260520-0-fe`、`新增登录功能-20260520-0-be`。
  - 全局唯一，出现在任务看板上，`-fe`/`-be` 后缀区分前后端。

## 任务粒度

- 以**功能模块的一次变更**为一个开发任务单元：架构师每完成一次模块设计或变更（对应 `architecture/module/<模块>/change-log/` 中的一条变更日志），按变更影响面决定分配哪些侧的任务。
- **不在架构师侧做实现级拆分**（如按 API 端点、按页面）——这是开发者的职责，在各开发者 Agent 的工作流中细化：前端 Agent 在开发流程中将任务拆分为组件/页面/接口调用等编码任务，后端 Agent 拆分为控制器/服务/数据访问等编码任务。
- 按下表选择分配方式：

| 变更类型 | 分配任务 | 说明 |
|---|---|---|
| 前后端并行（默认） | `-fe` + `-be` | 同时影响前后端，契约稳定，可并行启动 |
| 仅前端 | 仅 `-fe` | 纯 UI/文案/交互调整，不涉及 API 或 DB |
| 仅后端 | 仅 `-be` | DB 迁移、性能优化、索引调整、内部重构，不影响 API 契约或前端 |
| 前后端串行 | `-fe` + `-be`，备注标注依赖 | 破坏性 API 变更，前端须基于新契约开发，在后端完成契约变更后启动 |

- 无论分配几个任务，同一次模块变更共享同一个变更ID；变更日志照常生成。

## 流程

- 为本次模块变更生成**变更ID**：`<任务简介>-<YYYYMMDD>-<序号>`。
  - **任务简介**：对本任务所对应模块变更的简短描述（建议 5-15 字，**不得包含连字符 `-`**），如"新增登录功能"、"用户列表分页优化"。
  - **日期-序号**：与 `architecture/module/<模块>/change-log/` 下对应的变更日志文件名完全一致（如变更日志为 `20260520-0.md`，则变更ID 为 `新增登录功能-20260520-0`）。
- 按"任务粒度"小节的决策表确定本次分配哪一侧的任务，为每个待分配侧生成**任务编号**：`<变更ID>-fe` 或 `<变更ID>-be`。
- 任务看板字段填写规范：
  - **任务编号**：按上述格式生成（带 `-fe`/`-be` 后缀）。
  - **所属模块**：与 `/requirements/module/` 下的模块文件夹名一致。
  - **描述**：20 字以内的简述，说明本任务对应的模块变更（如"登陆鉴权模块新增登录功能"）。
  - **状态**：新任务统一为 `todo`。
  - **备注**：可选，填写额外说明（如跨模块依赖、串行依赖、注意事项）。任务编号已与变更日志编号绑定，无需重复填写。
- **幂等校验**：追加任务前，先读取目标看板（`/dev-tasks/frontend.md` 或 `/dev-tasks/backend.md`），检查是否已存在相同**任务编号**的行。若存在，跳过追加并报告"任务 `<编号>` 已存在，跳过"；若不存在，再追加。同一变更ID 下另一侧的任务编号不同，正常追加。
- 前端开发任务追加到 `/dev-tasks/frontend.md`，后端开发任务追加到 `/dev-tasks/backend.md`。如果没有任务看板请新建，任务看板模板请参考 `/templates/开发任务看板模板.md`。多个任务分行追加，不要覆盖已有任务。
