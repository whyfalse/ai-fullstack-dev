## 任务粒度

- 以**功能模块的一次变更**为一个开发任务单元：架构师每完成一次模块设计或变更（对应 `architecture/module/<模块>/change-log/` 中的一条变更日志），为前端和后端各分配一个任务。
- **不在架构师侧做实现级拆分**（如按 API 端点、按页面）——这是开发者的职责，在各开发者 Agent 的工作流中细化：前端 Agent 在开发流程中将任务拆分为组件/页面/接口调用等编码任务，后端 Agent 拆分为控制器/服务/数据访问等编码任务。
- 一个模块变更 → 一个前端任务 + 一个后端任务，两者可并行启动（前端基于 API 契约开发，后端基于 DB 与契约开发）。

## 流程

- 为前/后端开发任务分配任务id，格式为：`模块名称-日期-序号`，例如 `登陆鉴权-20260520-0`。
  - **模块名称**：与 `/requirements/module/` 下的模块文件夹名一致。
  - **日期-序号**：与 `architecture/module/<模块>/change-log/` 下对应的变更日志文件名完全一致（如变更日志为 `20260520-0.md`，则任务编号为 `<模块名称>-20260520-0`），保证任务与设计变更一一可追溯。
- 任务看板字段填写规范：
  - **任务编号**：按上述格式生成。
  - **描述**：20 字以内的简述，说明本任务对应的模块变更（如"登陆鉴权模块新增登录功能"）。
  - **状态**：新任务统一为 `todo`。
  - **备注**：可选，填写额外说明（如跨模块依赖、注意事项）。任务编号已与变更日志编号绑定，无需重复填写。
- 前端开发任务追加到 `/dev-tasks/frontend.md`，后端开发任务追加到 `/dev-tasks/backend.md`。如果没有任务看板请新建，任务看板模板请参考 `/appendix/开发任务看板模板.md`。多个任务分行追加，不要覆盖已有任务。
