##  核心原则
- **契约先行**：任何涉及前后端交互的需求，必须先定义或修改 API 契约。
- **模块划分**：按业务边界和功能独立性进行模块拆分，明确各模块的职责边界，避免跨层耦合。
- **影响优先**：改动前必须评估对数据库、接口、缓存、第三方服务及前端的影响范围。
- **可执行性**：分配给前后端的任务必须具体、可验证，包含明确的输入输出和验收标准。
- **安全与性能**：在设计阶段就考虑鉴权、限流、数据校验、索引优化等非功能需求。
- **数据独立性原则**：数据库设计独立于前端展示需求，基于业务实体关系建模。前端需要的数据格式通过接口层转换，不直接暴露原始表结构。
- **可演进性原则**：接口需有版本化策略（路径版本/头版本/媒体类型版本，由架构师按项目所选协议风格选定），允许增量变更；数据库设计预留扩展字段（如关系型的 `extra` JSON 列、文档型的动态字段），避免频繁改表/改集合结构。
- **反模式警告**：避免前后端共享同一个实体类/数据结构；避免在数据库里存储JSON格式的业务核心字段；避免接口返回数据库全部字段
## 注意

- 请务必遵守下面的执行步骤进行需求分析，不要擅自更改目录结构或流程。
- 设计顺序遵循"数据独立性原则"：先基于业务实体设计数据库，再设计 API 契约，最后设计前后端程序功能。不得因 API 需求反向驱动数据库设计。

## 执行步骤

- 读取需求并在`/requirements/module/`下找到对应的需求模块，理解功能细节、页面设计、验收标准、其他（安全性、质量要求等）。

- 若 `/architecture/README.md` 不存在或内容为空，初始化架构设计说明（包含架构设计思维、前后端技术选型、目录结构约定）；若已存在，仅在技术选型或目录结构发生重大变更时更新。

- 该需求可以划分到新的程序模块还是对原有模块进行更改，如果是新模块，创建`/architecture/module/<模块名称>/`文件夹，并在`/architecture/module/catalog.md`中追加一行（模块名称、简介、备注、状态）。新模块状态填 `active`。若是修改已有模块，按需更新 catalog 中对应行的简介；若本次涉及废弃见后续"废弃流程"步骤。

- 读取`/architecture/contracts/database/`下现有的数据库定义，理解当前业务实体模型，作为数据库表设计的参考。

- 识别业务实体，设计数据库表结构（基于业务实体关系建模，不预先考虑 API 需要返回什么字段）。

- 读取`/architecture/contracts/apis/`下现有的api定义，分析是否需要新增/修改 API。

- 识别前端、后端程序功能，并列出功能清单（前端：页面/组件/状态管理/API 调用；后端：控制器/服务/数据访问），作为后续 frontend.md / backend.md 的输入。

- 基于 DB 设计和功能清单，设计 API 契约（请求/响应格式、错误码、鉴权方式）。响应字段按前端需要裁剪，不直接暴露 DB 全部字段。

- 输出设计概要（包含模块归属、DB 设计、API 设计、程序逻辑、变更等信息），人工确认，若通过，继续后续流程，未通过则根据反馈从对应步骤重新分析。

- 将变更后的数据库表/集合写入到`/architecture/contracts/database/`下，统一使用 `.json` 中立描述（字段、类型、约束、索引的元数据，由开发者按目标数据库类型映射为具体 DDL 或 schema 验证），并更新目录`/architecture/contracts/database/catalog.md`。示例请参照`/templates/database-contract-template/users.json`，注意，严格按照示例文件的格式进行编写，不得额外添加内容。

- 若本次设计涉及数据库 schema 变更（新增表/改字段/删字段/改索引/改约束/废弃表），为**每张受影响的表**在`/architecture/contracts/database/change-log/<表名>.md`下追加一条变更记录。文件不存在则按`/templates/database-contract-template/change-log-template.md`的格式新建；已存在则在文件顶部（倒序）追加新条目。每条记录须包含：**变更类型**、**破坏性变更**（是/否及说明）、**变更详情**（字段/索引级逐条列出）、**迁移说明**（无则填"无"）、**关联模块变更日志**（指向本次将生成的模块 change-log 路径，命名按 dev-task ID 规则`<模块名称>-<YYYYMMDD>-<序号>`）。本次未涉及 schema 变更的模块跳过此步。

- 将新api接口更新到`/architecture/contracts/apis/interfaces/`，每个接口对应一个文件，将接口出入参定义更新到`/architecture/contracts/apis/params/`下，每个数据模型对应一个文件。最后更新`/architecture/contracts/apis/catalog.md`目录，接口与数据模型表格的"状态"列对新接口填 `active`。示例请参照`/templates/api-contract-template/`文件夹下对应文件，注意：严格按照示例文件的格式进行编写，不得额外添加内容。

- 生成前端程序功能，细化到页面/组件/样式/状态管理/API 调用，注意：不要写具体的实现代码。

- 将前端程序功能写入到`/architecture/module/<模块名称>/frontend.md`

- 生成后端程序功能，细化到控制器/服务/数据访问/单元测试，注意：不要写具体的实现代码。

- 将后端程序功能写入到`/architecture/module/<模块名称>/backend.md`。

- 生成功能测试用例，覆盖正常路径、边界场景和异常场景，明确输入数据与预期结果，注意：不要写具体的实现代码。

- 将功能测试用例写入到`/architecture/module/<模块名称>/test.md`。

- 若本次设计涉及**废弃**（接口废弃 / frontend.md/backend.md 部分功能废弃 / 模块整体废弃），按以下流程处理；本次未涉及废弃则跳过本步：
  - **接口废弃**（原地标注，不移动文件）：
    - 在 `/architecture/contracts/apis/catalog.md` 中将对应接口行的"状态"列改为 `deprecated`（永久下线改为 `archived`）；若该接口独有的数据模型不再被任何活跃接口引用，数据模型表对应行同样改状态。
    - 在对应 `interfaces/<接口名称>.md`（及 `params/<数据模型名>.md`）文件**顶部**追加废弃横幅，格式严格如下（占位项按实际填写）：
      ```
      > **已废弃**（<dev-task ID>，废弃日期 <YYYY-MM-DD>）：<废弃原因>。详见 [模块变更日志](../../../module/<模块名称>/change-log/<YYYYMMDD>-<序号>.md)。
      ```
    - 不得删除接口/数据模型文件本身，也不得移动到其他目录——历史 change-log 与 catalog 的相对链接必须保持有效。
  - **frontend.md / backend.md 部分功能废弃**：
    - 在对应文件末尾追加或更新 `## 已废弃` 小节，条目格式：`- <功能名称>（<dev-task ID>，<YYYY-MM-DD>）：<废弃原因>。详见 [模块变更日志](./change-log/<YYYYMMDD>-<序号>.md)。`
    - 不删除原功能描述正文，保留可追溯；对应模块 catalog 行状态改为 `deprecated`。
  - **模块整体废弃**（归档移动）：
    - 将 `/architecture/module/<模块名称>/` 整体移动到 `/architecture/module/_archived/<模块名称>/`，模块下的 frontend.md/backend.md/test.md/change-log/ 原样保留作为历史档案。
    - 在 `/architecture/module/catalog.md` 中将对应行的"状态"列改为 `archived`，保留行不删除。
    - 模块涉及的接口按"接口废弃"流程在原位标注；涉及的表按 DB change-log "废弃表"变更类型记录（已在数据库变更日志步骤覆盖）。
  - **模块 change-log 记录**：本次模块 change-log（模块整体废弃时为归档前最后一条）须包含一个**废弃内容**小节，列出废弃的接口、表、功能点及 dev-task ID，与各废弃文件顶部的横幅构成双向引用。

- 变更内容写入到`/architecture/module/<模块名称>/change-log/<日志编号>.md`，编号规则为日期+序号（例：20260522-0）。变更内容须包含一个**数据库变更**小节：若本次涉及 schema 变更，列出每张受影响表名、变更类型、是否破坏性，并给出指向`/architecture/contracts/database/change-log/<表名>.md`的链接；若未涉及 schema 变更，注明"无"。该小节与数据库 change-log 构成双向引用，dev-task ID 保持一致。若本次涉及废弃，还须包含上一步所述的**废弃内容**小节。

- 对照核心原则进行自检，发现问题需回到对应步骤修正：
  - **契约先行**：所有前后端交互是否都有对应的 API 契约定义？
  - **数据独立性**：DB 表是否基于业务实体建模，而非按 API 响应字段反推？
  - **可演进性**：API 是否定义了版本化策略（路径/头/媒体类型之一）？DB 是否预留扩展字段？
  - **反模式**：是否避免了前后端共享实体类？API 响应是否裁剪而非返回 DB 全字段？是否避免了 DB 存储 JSON 业务核心字段？
  - **变更可追溯**：涉及 schema 变更的每张表是否都已在 `/architecture/contracts/database/change-log/<表名>.md` 追加条目，且与模块 change-log 双向引用、dev-task ID 一致？
  - **废弃可追溯**：废弃的接口/数据模型是否在 `apis/catalog.md` 标注状态、文件顶部追加废弃横幅并指向模块 change-log？frontend.md/backend.md 部分废弃是否归入 `## 已废弃` 小节？模块整体废弃是否已移至 `module/_archived/<模块名称>/` 且 catalog 状态改为 `archived`？模块 change-log 是否包含"废弃内容"小节？

- 总结出变更概览。
