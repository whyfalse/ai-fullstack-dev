## 强制要求：注释规范

本项目的代码会被 AI 开发者反复读取与修改，注释是 AI 理解项目意图的主要线索，以下为**强制**要求：

- **文件头注释**：每个脚本文件顶部必须有一段文件头注释，说明文件职责、所属模块、对外暴露内容。
- **函数注释**：每个非私有函数必须有职责、参数、返回值、可能异常的说明。
- **关键行注释**：以下场景必须加行内注释——业务规则相关的魔法数字、绕过框架默认行为的 hack、性能优化的特殊写法、与后端契约绑定的字段映射、临时方案/TODO。
- **注释随代码演进**：修改代码必须同步修改注释，删除代码必须连同注释删除。过时注释比无注释更危险。
- **不写废话注释**：不重复代码已表达的内容，只解释"为什么"和"非显而易见的约束"。

## 易错场景约束

### 抽象时机
- **三次法则**：同样逻辑写第三次前才抽取，前两次容忍重复。提前抽取会产生错误抽象，比重复危害更大。
- **复用是结果不是目标**：不要为"未来可能复用"提前抽公共组件。两段相似但分属不同业务模块的代码，保持独立比强行共享更易演进。

### 分层与副作用
- **容器与展示分离**：展示组件不直接调 API、不感知全局状态。副作用（请求、定时器、localStorage）收口在 Hook 或服务内，组件主体保持纯净。
- **契约校验在边界**：在 API 响应、props、URL 参数的入口处校验一次，内部代码信任已校验数据，不层层重复校验。

### 状态
- **状态最小化**：能派生的不存状态，能用 URL 表达的不存内存，能由服务端返回的不存本地。
- **状态就近**：状态放在使用它的最近组件节点，只在多兄弟共享时上提，不要一开始就用全局状态。
- **不可变更新**：状态更新返回新对象，不原地 mutation。

### 变更范围
- **最小化变更**：修改既有代码只改必须改的部分，不顺手重构。重构应有独立任务。
- **废弃不删除**：暂时不用的代码标记 `@deprecated` 并注明替代方案与计划下线时间，给调用方迁移窗口。

### 注释意图历史
- **保留决策痕迹**：重要的设计决策、绕路方案、临时妥协在注释中记录原因。半年后你自己也记不清为什么这么写。

## 自检清单

提交前对照检查：

- [ ] 文件头、函数、关键行注释齐全且与代码同步
- [ ] 重复三次以上的逻辑才抽取，无提前抽象
- [ ] 副作用已收口，组件主体纯净
- [ ] 状态最小化，无派生状态、无冗余全局状态
- [ ] 既有代码只做必要变更，未顺手重构
- [ ] 边界校验在入口，内部不重复防御
- [ ] 废弃代码已标记 `@deprecated` 而非删除
